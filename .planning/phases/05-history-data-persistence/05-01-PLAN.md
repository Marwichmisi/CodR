# Plan 05-01: SQLite history storage integration and history list view

## Context

Phase 1 created `StorageService` with full CRUD for `scan_records` and `generation_records` tables, plus `ScanRecord` and `GenerationRecord` models. However, **nothing in the app calls StorageService** — scans and generations are never persisted. `HistoryScreen` is a static placeholder (icon + title, no ViewModel). This plan wires the persistence layer to existing screens and builds the history display.

Key decisions from CONTEXT.md: D-01 (RecordBase abstraction), D-03 (getHistory method), D-04/D-05/D-06/D-07 (FIFO strategy), D-08/D-09/D-10/D-11/D-12 (StorageService injection), D-13/D-14/D-15 (list item layout).

## Tasks

**Action:**

1. Create `lib/models/record_base.dart` (per D-01, D-02):
   - Define `abstract class RecordBase` with getters: `int get id`, `String get content`, `DateTime get timestamp`, `String get type`
   - Add a static `factory RecordBase.fromJson(Map<String, dynamic> json)` that reads `json['type']` and delegates to `ScanRecord.fromJson(json)` or `GenerationRecord.fromJson(json)`
   - Follow `flutter-implement-json-serialization` skill for model pattern

2. Modify `lib/models/scan_record.dart`:
   - Import `record_base.dart`
   - Change `class ScanRecord` to `class ScanRecord extends RecordBase`
   - Mark the existing `fromJson` factory as `@override`

3. Modify `lib/models/generation_record.dart`:
   - Import `record_base.dart`
   - Change `class GenerationRecord` to `class GenerationRecord extends RecordBase`
   - Mark the existing `fromJson` factory as `@override`

4. Add `getHistory()` to `lib/services/storage_service.dart` (per D-03):
   - New method `Future<List<RecordBase>> getHistory()` that:
     - Calls `getAllScanRecords()` and `getAllGenerationRecords()`
     - Merges into `List<RecordBase>`
     - Returns sorted by `timestamp DESC`

5. Add FIFO cleanup to `lib/services/storage_service.dart` (per D-04, D-05, D-06, D-07):
   - New private method `_enforceFifoLimit(Database db, String table, {int maxRecords = 100})`
   - Logic: COUNT(*) from table, if > maxRecords, DELETE oldest via `WHERE id IN (SELECT id FROM table ORDER BY timestamp ASC LIMIT (count - maxRecords))`
   - Call `_enforceFifoLimit` inside both `insertScanRecord` and `insertGenerationRecord` after insert (per D-07: only on insert, not at app start)
   - Per D-05: FIFO is **separate per type** (max 100 scans AND max 100 generations independently)

6. Create `test/models/record_base_test.dart`:
   - Test that `ScanRecord` and `GenerationRecord` are subtypes of `RecordBase`
   - Test `RecordBase.fromJson` factory returns correct subtype based on `type` field
   - Test shared getters (id, content, timestamp, type) work on both subtypes via RecordBase reference

**Acceptance Criteria:**
- [ ] `ScanRecord` and `GenerationRecord` both extend `RecordBase`
- [ ] `RecordBase.fromJson()` correctly returns `ScanRecord` when type='scan' and `GenerationRecord` when type='generation'
- [ ] `StorageService.getHistory()` returns merged list sorted by timestamp DESC
- [ ] After inserting 101 records into `scan_records`, only 100 remain (oldest deleted)
- [ ] After inserting 101 records into `generation_records`, only 100 remain (oldest deleted)
- [ ] FIFO limit for scans does NOT affect generations count (separate per type, per D-05)
